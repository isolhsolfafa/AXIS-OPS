# CT "데이터 신뢰도" 지표 방법론 검증 — Fable 교차검증 brief (2026-06-13)

> 목적: GST 제조 CT 분석 페이지 "데이터 신뢰도" 2개 지표의 **방법론 타당성 + 기준(granularity) 정의** 독립 판정.
> Claude(Opus) 분석 + Codex(GPT) 1차 검증 완료 → **Fable 3차 독립 판정** 요청. 코드 fix 아님, 지표 설계 권고.
> ⚠️ Fable 승격 트리거: 다중 제약 trade-off 의사결정 + 방법론 판단 (CLAUDE.md 리드 섹션).

## 1. 배경 — 무엇을 재는 지표인가

GST는 제조 현장 작업을 OPS app으로 태깅(작업 시작/완료)하고, 그 duration으로 **CT(cycle time) 표준**을 산출하려 함. 최종 목표 = APS-Lite(생산계획). CT 표준은 **모델별·협력사별·공정별**로 적용됨.

"데이터 신뢰도" 섹터 2개 지표:

### 지표 A — "CT 신뢰도" (표본 가중)
- 산식: `Σ(weight(task표본) × task표본) ÷ Σ(task표본)`, weight = 100건+ →1.0 / 30~99 →0.5 / 30미만 →0.1
- 의미: "task별 표본이 CT 표준 삼을 만큼 충분한가"를 표본 수로 가중 평균

### 지표 B — "입력 정합" (duration_source clean%)
- 산식: `clean / total`, clean = duration_source ∈ (NULL, NORMAL_COMPLETION)
- 의미: "마감이 정상완료인가(추정 cap/fallback 아님)"

## 2. 실데이터 측정 결과 (운영 DB, 2026-06-13)

### 지표 A — granularity 별 (쪼갤수록 무너짐)
| 단위 | 신뢰도 | 100건+ 셀(가중 1.0) | 셀 수 |
|---|---|---|---|
| task 전체 풀링 (**현 헤드라인**) | **80.6%** | 6개 | 15 |
| 협력사 × task | 40.4% | **0개** | 28 |
| **협력사 × 모델 × task** (실사용 단위) | **13.1%** | **0개** | 117 |

- 핵심: weight 계단식(100/30 경계) → **풀링하면 경계 넘어 가중↑** → 최대 풀링이 최댓값. 실사용 단위(협력사×모델)로 가면 100건+ 셀 0개 → 13%.

### 지표 B — duration_source 분포
```
NULL                 77.5%  clean=True   ← ⚠️ force-close 포함 (duration_source 미설정→NULL)
NORMAL_COMPLETION     3.2%  clean=True
PREV_DAY_CAP         18.1%  clean=False
FALLBACK_17           1.2%  clean=False
→ clean% = 80.7%
```

### 배경 제약 (문서화됨, memory ADR)
app 도입 초기 태깅 미성숙: 제품당 완료 태깅 **22%**, MECH 실시간 추적 **9%**, 실작업 70~80% 미태깅. ADR = "product-level CT/효율/lead/납기 = 시기상조, 해석 금지".

## 3. Claude(Opus) 분석 결론
1. 80.6%는 "최대 풀링" 낙관값 — 실사용 단위(협력사×모델)는 13%.
2. 표본 "수"만 보지 품질(추적율 9%) 미반영 → "100건 있어도 70% 미태깅".
3. 지표 B의 NULL=clean이 force-close 포함 → 입력정합 부풀림.
4. A(표본 수)·B(source clean)는 다른 축이나 **둘 다 부분 품질축을 "데이터 신뢰도"로 확대 표현하는 같은 함정**.

## 4. Codex(GPT) 1차 검증 결론 (Claude와 합의)
- **M**: 헤드라인 80.6%는 오도 위험 — 실사용 단위(협력사×모델 13%)로 잡거나 라벨 "전체 task 표본 충분도"로 좁혀야.
- **M**: 표본 수만으로 합성은 결함 — coverage bias(미태깅 70~80% 편향=생존자 편향) 미반영. `표본충분도 × 추적율` 2축 분리 권고.
- **M**: 지표 B NULL=clean force-close 포함 = **분류 로직 결함**(force_closed=FALSE 가드 추가 + FORCE_CLOSED_MISSING_SOURCE bucket 분리).
- **A**: 0초 duration plausibility는 별도 값-타당성 축 분리.
- **권고**: 라벨 정정(CT 신뢰도→표본 충분도 / 입력정합→duration_source 정합도) + 보수 단위(협력사×모델) + 품질축(추적 커버리지) 추가 + 해석 게이트(추적율 임계 미달 시 CT/효율/납기 해석 금지 표시).

## 5. Fable 독립 판정 요청 (5개)

> Claude·Codex가 합의했으나 **맹목 동조 방지** 위해 Fable이 독립적으로 재판단. 두 모델이 틀렸거나 놓친 게 있으면 반박.

1. **헤드라인 granularity**: "CT 신뢰도" 헤드라인을 협력사×모델(13%)로 잡는 게 정말 맞나? "통합 CT 표준"이 유효한 사용처(예: 공정 난이도 벤치마크)가 있으면 풀링(80%)도 헤드라인 자격이 있는지. 13% 헤드라인이 오히려 과소 비관(데이터 충분히 쌓인 task까지 묻어버림)일 위험은 없는지.
2. **2축 분리 vs 단일 합성**: 표본충분도 × 추적율 2축이 정말 더 정직한가, 아니면 사용자(현장 관리자)에겐 단일 점수가 의사결정에 더 유용한지. 2축이 "분석마비" 유발 위험.
3. **weight 계단식 자체**: 100/30 경계 계단이 풀링 편향의 근본 원인. 연속 함수(예: min(1, n/100)) 또는 베이지안 shrinkage로 바꾸면 풀링 편향이 완화되는지, 아니면 계단이 현장 직관에 더 맞는지.
4. **지표 B force-close 가드**: force_closed 제외가 입력정합을 "정정"하는 게 맞나, 아니면 force-close도 "마감은 됨"이라 입력 관점에선 포함이 맞는지(CT 신뢰도와 입력정합의 목적 차이). 0초 duration을 입력정합이 봐야 하는지 CT 신뢰도가 봐야 하는지 경계.
5. **현 단계 최선의 헤드라인 1문장**: 태깅 미성숙(추적 9%) 상황에서 현장 관리자에게 "데이터 신뢰도"를 한 줄로 정직하게 표현한다면? Codex 제안("협력사×모델 표본충분도 13.1% + 추적 커버리지 9%로 해석 제한")이 최선인지, 더 나은 표현이 있는지.

### 판정 형식
- 각 질문 M(설계 결함, 반드시 정정) / A(권고) / N(현행 타당) 라벨 + 근거.
- Claude·Codex 합의에 대한 **동조/반박** 명시 (맹목 동조 금지, CLAUDE.md L98/L149).
- 최종 권고: 헤드라인 정의 + 라벨 + 품질축 구성을 1개 안으로 수렴.

## 부록 — 산식 재현 코드 (검증용)

```python
def weight(n): return 1.0 if n>=100 else 0.5 if n>=30 else 0.1
# CT 신뢰도 = Σ(weight(task_n)*task_n) / Σ(task_n)
# 단위: task 풀링 / 협력사×task / 협력사×모델×task (granularity)
# 입력정합 = (NULL + NORMAL_COMPLETION) / total  [NULL에 force-close 포함됨]
```
- 소스: `backend/app/services/statistics_service.py`(get_task_ct_stats / get_partner_breakdown / get_data_quality) + VIEW `CtReliabilityTrendCard.tsx` L17/63~78.
- 운영 실측: task 풀링 80.6% / 협력사×task 40.4% / 협력사×모델×task 13.1% / 입력정합 80.7%.




Fable 독립 판정 — CT "데이터 신뢰도" 지표 방법론
먼저 전체 입장: Claude·Codex 합의는 방향이 옳습니다. 하지만 한 곳에서 과교정(13% 헤드라인)이고, 더 깊은 근본 문제 하나를 둘 다 놓쳤습니다. 질문별로 갑니다.
Q1. 헤드라인을 협력사×모델(13%)로? → A (권고) — 합의에 부분 반박
13%를 헤드라인으로 박는 건 반대합니다. Claude·Codex가 "최대 풀링 80%는 낙관"이라 본 건 맞지만, 반대 극단인 협력사×모델 13%도 똑같이 오도입니다 — 데이터가 충분히 쌓인 task(예: GST 마무리공정 n=40)까지 13% 안에 묻어버리니까요. 이건 본인이 질문에 스스로 적은 "과소 비관" 우려가 정확합니다.
근본 오류는 단일 헤드라인 숫자를 강요하는 것 자체입니다. 신뢰도는 사용처(granularity)에 종속적입니다 — "통합 공정 벤치마크"용이면 풀링 80%가 유효하고, "협력사별 표준 적용"용이면 13%가 진실입니다. 같은 데이터의 다른 질문이죠. 권고: 헤드라인을 숫자 1개가 아니라 "적용 단위별 충분도" 미니 표로 (풀링 81% / 협력사×task 40% / 협력사×모델 13%). 이미 군별/협력사별 토글이 그 구조이니, 헤드라인만 거기에 맞추면 됩니다.
Q2. 2축 분리(표본충분도 × 추적율)? → N (현행 합의 타당) — 동조, 단 표현 조건부
2축이 맞습니다. 표본 "수"와 추적 "품질"은 정말 다른 축이고, 합성하면 "100건인데 70% 미태깅"이 가려집니다 — 이건 통계적으로 교정 불가한 생존자 편향이라 합성이 거짓말이 됩니다. "분석마비" 우려는 이해하지만, 현장 관리자에게 줄 건 2개 숫자가 아니라 2축으로 계산된 게이트 1개입니다: 추적율이 임계 미달이면 충분도와 무관하게 "해석 보류" 신호. 즉 내부는 2축, 표면은 신호등 1개. 이러면 정직성과 단순성을 둘 다 얻습니다.
Q3. weight 계단식 → 연속/베이지안? → A (권고) — 둘 다 놓친 지점 보강
계단식(100/30)이 풀링 편향의 수학적 원인이라는 진단은 맞습니다. min(1, n/100) 연속 함수로 바꾸면 경계 점프가 사라져 풀링해도 갑자기 안 뛰니 편향이 완화됩니다 — 이건 채택할 만합니다.
그런데 Claude·Codex 둘 다 놓친 게 있습니다: 이 가중평균은 "표본이 많은 task"에 가중치를 더 줍니다. 즉 데이터가 많은 task일수록 점수를 끌어올리는데, 정작 표준이 부실해서 위험한 건 표본 적은 task입니다. 지표가 "안심해도 되는 곳"을 강조하고 "위험한 곳"을 희석하는 방향이에요. 베이지안 shrinkage보다 더 정직한 건 "표준으로 쓸 수 있는 task 비율" (n≥30 & 추적율≥임계인 task 수 ÷ 전체 task 수) 같은 count 기반 지표입니다 — 가중평균은 한 숫자에 너무 많은 걸 욱여넣습니다.
Q4. 지표 B force-close 가드? → M (결함, 정정) — 합의에 동조하되 경계 재정의
force_closed=FALSE 가드 추가는 맞습니다. 단 Claude·Codex가 "force-close도 마감은 됨"이라는 반론을 충분히 안 풀었는데, 제 답은: 지표 B의 이름이 "입력 정합"이면 force-close는 빠져야 합니다. force-close는 "작업자가 입력 안 해서 관리자가 대신 닫은 것" = 입력 실패의 증거지 정합의 증거가 아닙니다. 지금은 그게 NULL에 섞여 clean 80.7%를 부풀리는데, 실제로는 77.5% NULL 중 상당수가 force-close라 진짜 정상완료는 훨씬 적을 수 있습니다.
정정 방향은 Codex 제안(FORCE_CLOSED_MISSING_SOURCE 버킷 분리)이 정확합니다. 추가로 — 이건 데이터 모델 결함의 증상입니다(force-close가 duration_source를 NULL로 남김). 근본 fix는 force_close_task가 duration_source='FORCE_CLOSED'를 기록하게 하는 것이고, 그러면 지표 B도 CT 표본 정정(v2.36.1)도 가드 없이 자동 정합됩니다.
Q5. 현 단계 최선의 헤드라인 1문장? → 제안
Codex 안("협력사×모델 13% + 추적 9%")은 정직하지만 두 비관 숫자를 곱해 절망적으로 들립니다 — 관리자가 "이 시스템 못 쓰겠네"로 읽으면 태깅 동기가 오히려 꺾입니다. 현 단계 목적은 "CT를 믿어라"가 아니라 "태깅을 채워라"이므로, 헤드라인도 그 행동을 유도해야 합니다:

"실시간 추적 9% — 지금은 표준 산출이 아니라 태깅 정착 단계입니다. (참고: 데이터 쌓인 task 6개는 검증 가능, 협력사별 적용은 추적율 70%↑ 도달 후)"

추적율을 첫 줄에 단독으로, CT 충분도는 괄호 참조로. 이유: 추적율이 모든 것의 게이트이고(9%면 나머지 다 무의미), 관리자에게 "지금 할 일 = 태깅"을 직접 가리킵니다.

종합 권고 (1개 안으로 수렴)

헤드라인 = 추적 커버리지(9%) 단독 + "태깅 정착 단계" 라벨. CT 충분도는 종속 표시
CT 충분도 = 적용 단위별 미니 표(풀링/협력사×task/협력사×모델) — 단일 숫자 금지
라벨 정정: "CT 신뢰도"→"표본 충분도", "입력 정합"→"마감 출처 정합"
2축은 내부 계산, 표면은 해석 게이트 1개(추적율 임계 미달 시 CT/효율/납기 "해석 보류")
count 기반 보조지표 추가: "표준 가능 task 비율"(n≥30 & 추적≥임계)
근본 fix: force_close가 duration_source='FORCE_CLOSED' 기록 → 지표 B·CT 표본 둘 다 가드 없이 정합 (Codex 버킷 분리는 그 전 임시방편)

Claude·Codex 대비 차이: 방향(2축·라벨·force-close 가드·해석 게이트)은 동조. 반박/보강 3건 — ①13% 헤드라인 반대(과소 비관, 적용단위 표로) ②가중평균이 "위험한 task를 희석"하는 구조적 결함(count 지표로 보완) ③force-close는 증상이고 근본은 데이터 모델(duration_source 미기록)이라 거기서 막아야 함.

## Fable 독립 판정 요청 (5개 — M/A/N + 근거)
1. **count 게이트의 2조건 AND**: `n≥30 AND 추적≥70%` 동시 충족 셀 비율이 가중평균보다 정직한 품질게이트인 게 맞나? 현 단계 추적 9%면 둘 다 충족 셀이 거의 0 → 게이트가 0~5%로 수렴할 텐데, 이게 "정직"인지 아니면 **과도하게 비관(=가중평균 과대평가의 반대 극단)**인지. 1차에서 Fable이 13% 헤드라인도 과소비관이라 했는데, 2조건 AND는 그보다 더 낮음.
2. **종합 게이트 단위 = 모델×task (협력사 합산)**: 사용자는 "종합은 모델별, 협력사는 매트릭스 drill" 결정. 근데 실측상 같은 GAIA를 BAT/FNI가 분담 → 모델×task로 협력사 합산하면 BAT(빠름)+FNI(느림)이 한 셀에 섞임. 이게 summary 종합용으론 OK인가, 아니면 협력사×모델×task 가 게이트로도 맞나?
3. **공정추적율 broadcast vs task-level**: Codex는 task별 tracked/n 으로 좁히라 권고. summary 게이트 정밀도에 task-level 가치가 있나, 아니면 공정 proxy 로 충분(현 단계 추적 미성숙이라 어차피 거친 신호)한지.
4. **가중평균 ref 유지 가치**: ct_sufficiency(가중평균)를 참고로라도 남길 가치가 있나, 아니면 혼란만 주니 완전 폐기? (사용자: 가중평균이 문제의 근원)
5. **현 단계 summary card 최선 1안**: 2 게이트(표준가능 비율 + 마감출처 정합) + 헤드라인(추적율) 구성이 현 단계(태깅 9%) 관리자에게 최선인지, 더 나은 1안이 있는지. 특히 "표준 가능 비율"이 거의 0%로 나올 때 관리자 행동(태깅 채우기)을 유도하는 표현.

Fable 2차 독립 판정 — B 구현 설계 (count 게이트)
전체 입장: 방향 전환(가중평균 폐기 → count 게이트)은 옳고, 1차 권고를 충실히 구현했습니다. 단 사용자 결정 1건(Q2)에 방법론적 반박이 있고, 1차와의 일관성을 위해 Q1에서 제 입장을 명확히 구분합니다.
Q1. 2조건 AND 게이트, 거의 0%로 수렴 → N (타당) — 1차와 모순 아님, 단 진행 신호 보강 필요(A)
먼저 1차와의 외견상 충돌을 정리합니다. 1차에서 13%를 반대한 건 그게 헤드라인이었기 때문입니다(데이터 쌓인 task까지 묻어버림). 지금 standard_ready_pct는 헤드라인이 아니라 게이트(헤드라인은 추적율). "표준으로 승격 가능한 셀 비율"이라는 질문에 대해 거의 0%는 과소비관이 아니라 정답입니다 — 추적 9%면 실제로 표준 삼을 셀이 없는 게 사실이니까요. 역할이 다르므로 모순이 아닙니다. 이건 명확히 해두겠습니다(거짓 일관성 피하려고).
다만 binary 게이트가 몇 주째 0%면 진행도(gradient)가 안 보입니다 — 태깅이 9%→20%로 올라도 게이트는 여전히 0이라 "나아지고 있다"가 안 잡혀 동기를 꺾습니다. 보강: 게이트를 0/1이 아니라 2조건을 분리 표시(n≥30 충족 셀 X% / 추적≥70% 충족 셀 Y% / 둘 다 = 표준가능 Z%). 그러면 한 조건이 먼저 차오르는 게 보입니다. 게이트 개념은 N, 진행 신호는 A.
Q2. 종합 단위 = 모델×task (협력사 합산) → A (강한 권고) — 사용자 결정에 반박
여기가 핵심 쟁점입니다. 사용자는 "종합은 모델별, 협력사는 매트릭스 drill"로 결정했는데, 게이트 단위로는 위험합니다. 이유는 본인이 §1.3에 적은 실측 그대로입니다 — GAIA를 BAT(즉시완료 61%)와 FNI(11%)가 분담하는데, 모델×task로 협력사를 합치면:

n_raw가 협력사 합산이라 n≥30 문턱을 더 쉽게 넘습니다 → 게이트가 "표준 가능"이라 말하기 쉬워짐
그런데 실제 표준은 협력사×모델로 적용됨(본인 결론: "진짜 적용단위 = 협력사×모델×task")
결과: 게이트가 "준비됨"이라 했는데 FNI엔 적용 불가한 표준을 출하 — 가중평균 과대평가에서 벗어나려다 같은 방향(낙관 편향)의 다른 버그를 만듦

게이트의 철칙은 **"표준이 소비되는 단위 = 게이트가 검사하는 단위"**입니다. 적용이 협력사×모델이면 게이트도 협력사×모델이어야 합니다. UX상 summary를 모델별로 보여주는 건 좋지만(드릴 구조 유지), 합격/불합격 판정 계산은 협력사×모델 셀에서 하고 모델 레벨엔 "그 모델의 협력사×모델 셀 중 표준가능 비율"로 rollup하는 게 맞습니다. 모델별 표시 ≠ 모델별 판정. 이 둘을 분리하면 사용자 UX 결정과 게이트 정합성을 둘 다 지킵니다.
Q3. broadcast vs task-level 추적율 → N (Codex A 기각) — 현 단계 과잉정밀
Codex가 "task별 tracked/n으로 좁히면 더 정밀"이라 한 건 원론적으론 맞지만, 추적 9% 단계에선 노이즈입니다. 공정 proxy든 task-level이든 둘 다 "거의 다 미달"로 나옵니다. task-level 정밀화는 추적율이 50%+ 올라와 셀 간 차이가 의미를 갖기 시작할 때 도입할 가치가 생깁니다. 지금 넣으면 코드 복잡도만 늘고 신호는 동일. 보류 권장(Codex의 A를 N으로 하향).
Q4. 가중평균 ref 유지 → A — 카드에서 완전 제거, 서버 전용으로만
사용자가 "가중평균이 문제의 근원"이라 했는데, ref로 접어둬도 누군가는 펼쳐서 80%를 인용합니다 — 본부 보고에 그 숫자가 한 번 들어가면 정정 비용이 큽니다. 1차에서 제가 "단일 숫자 강요가 오류"라 한 것의 연장: 카드에서 빼세요. 추세 비교용으로 값이 필요하면 서버 응답에만 두고 FE 렌더는 하지 않는 게 안전합니다("계산은 하되 보여주지 않음"). ref 접기 = 여전히 노출.
Q5. 현 단계 summary card 최선 1안
2 게이트 + 헤드라인 구성은 맞습니다. "표준가능 비율 0%"의 동기부여 문제는 헤드라인이 해결합니다 — 추적율(9%)이 gradient를 가진 행동 지표라 "이걸 올려라"가 명확하니까요. 표준가능 비율은 "아직 멀었다"의 정직한 게이트로 두되, 0% 자체가 절망이 아니라 "추적율 올리면 자동으로 차오른다"는 인과를 한 줄로 묶으면 됩니다:

헤드라인: 실시간 추적 9% — 태깅 정착 단계 (해석 보류)

게이트 1 · 표준 가능: 0% (협력사×모델 셀 중 n≥30 & 추적≥70%) — 추적율이 오르면 차오릅니다
